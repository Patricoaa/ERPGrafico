# ADR-001: Decisiones de Arquitectura Fundacionales — ERPGrafico Frontend

**Estado:** Activo  
**Fecha:** 2026-03-31  
**Autores:** Equipo ERPGrafico  
**Impacto:** Todo el frontend

> Este ADR documenta las decisiones técnicas fundacionales ya implementadas y en producción.
> **No están abiertas a debate.** Para cambiar cualquiera de estas decisiones, se debe redactar un nuevo ADR que lo sustituya explícitamente y sea aprobado por el equipo.

---

## Contexto

ERPGrafico es un sistema ERP B2B con múltiples módulos (Ventas, Inventario, Producción, Tesorería, Compras, RRHH, Contabilidad, Contactos). El frontend requiere una arquitectura que soporte:
- Alta densidad de información (tablas, formularios complejos, dashboards).
- Consistencia visual estricta entre módulos.
- Desarrollo ágil con LLMs (vibe coding) sin sacrificar calidad de código.
- Escalabilidad modular sin acoplamiento entre features.

---

## Decisiones

### 1. Framework: Next.js (App Router)

**Decisión:** Next.js con App Router como framework principal.

**Razones:**
- Server Components para páginas de solo lectura (dashboards, listados) — reduce bundle size y mejora Time to First Byte.
- Client Components solo donde hay interactividad real (formularios, modales, tablas con filtros).
- Routing basado en sistema de archivos, alineado con Feature-Sliced Design.

**Consecuencias:**
- Los componentes en `app/` son Server Components por defecto. Agregar `'use client'` solo cuando sea estrictamente necesario.
- No usar `useEffect` para data fetching en Server Components — usar `async/await` directamente.
- **No reemplazar por:** Vite + React SPA, Remix, ni ningún otro framework sin nuevo ADR.

---

### 2. Lenguaje: TypeScript con Zero Any Policy

**Decisión:** TypeScript estricto. Prohibición absoluta de `any`.

**Razones:**
- El ERP maneja lógica financiera y de inventario donde un tipo incorrecto puede traducirse en error de negocio real.
- Permite que el LLM genere código con contratos verificables en tiempo de compilación.
- `unknown` + type guards es el patrón correcto para datos de origen incierto.

**Consecuencias:**
- `tsconfig.json` debe tener `"strict": true` permanentemente.
- Los tipos de entidades de negocio se derivan siempre de Zod schemas (`z.infer<typeof Schema>`).
- **No reemplazar por:** JSDoc types, PropTypes, ni relajar a `strict: false`.

---

### 3. Sistema de Componentes: shadcn/ui

**Decisión:** shadcn/ui como biblioteca base de componentes UI.

**Razones:**
- Componentes copiados al proyecto (no dependencia de npm) — control total sobre el código.
- Accesibilidad (a11y) correcta por defecto vía Radix UI primitives.
- Compatible con Tailwind v4 y CSS variables para theming.
- El LLM conoce bien su API, reduciendo fricción en vibe coding.

**Consecuencias:**
- Los archivos en `/components/ui/` son el código fuente de shadcn — **no se modifican directamente**.
- Cualquier extensión o variante va en `/components/shared/` siguiendo `component-contracts.md`.
- **No reemplazar por:** MUI, Ant Design, Chakra UI, Mantine, ni ninguna otra librería de componentes.

---

### 4. Estilos: Tailwind CSS v4

**Decisión:** Tailwind CSS v4 como sistema de estilos, con CSS variables oklch como fuente de verdad de color.

**Razones:**
- Tailwind v4 usa `@theme inline` para mapear variables CSS a clases utilitarias — permite theming dinámico sin purge issues.
- oklch es el espacio de color más adecuado para sistemas de diseño modernos: perceptualmente uniforme, soporte de P3, fácil de manipular lightness/chroma.
- El sistema de tokens semánticos (`--success`, `--warning`, `--destructive`, `--info`) desacopla el color del significado.

**Consecuencias:**
- `globals.css` es la fuente de verdad visual. Ningún documento de diseño tiene mayor autoridad.
- Prohibido usar valores hardcoded de color en componentes. Solo clases Tailwind que mapeen a variables CSS.
- **No reemplazar por:** CSS Modules, styled-components, Emotion, ni Tailwind v3.

---

### 5. Tipografía: Onest + Syne

**Decisión:** Onest para body/UI, Syne para headings.

**Razones:**
- Onest: alta legibilidad en densidades de información elevadas, soporte de tablas numéricas, neutral sin ser genérica.
- Syne: carácter editorial y técnico para títulos, refuerza identidad industrial del sistema.
- Ambas disponibles en Google Fonts con subsetting adecuado.

**Consecuencias:**
- `font-sans` → Onest. `font-heading` → Syne. Configuradas en `globals.css` y `layout.tsx`.
- Los headings (`h1`–`h6`) tienen `font-heading font-extrabold tracking-tighter uppercase` aplicado globalmente.
- **No reemplazar por:** Plus Jakarta Sans, Inter, ni ninguna otra fuente sin nuevo ADR.

---

### 6. Data Fetching: TanStack Query (React Query)

**Decisión:** TanStack Query v5 para todo el data fetching del cliente.

**Razones:**
- Cache automático, invalidación, re-fetch en background y estados de loading/error listos para usar.
- Permite el patrón de hooks de feature (`use[Entity][Action]`) que encapsula `useQuery`/`useMutation` lejos de los componentes UI.
- Devtools integrados para debugging.

**Consecuencias:**
- `useQuery` y `useMutation` nunca se usan directamente en componentes UI — siempre dentro de hooks de feature en `src/features/[modulo]/hooks/`.
- Las query keys siguen el patrón `['entity-name', id?, filters?]` para invalidación precisa.
- **No reemplazar por:** SWR, Apollo Client, RTK Query, ni fetch manual con `useEffect`.

---

### 7. Validación y Schemas: Zod

**Decisión:** Zod como librería de validación y definición de tipos en runtime.

**Razones:**
- Un solo schema define el tipo TypeScript Y la validación en runtime — elimina duplicación.
- Integración nativa con `react-hook-form` vía `@hookform/resolvers/zod`.
- Permite validar y parsear respuestas del backend de Django de forma segura.

**Consecuencias:**
- Todo schema vive en `schema.ts` dentro del feature correspondiente.
- Los tipos de entidades se derivan con `z.infer<typeof Schema>` — nunca se definen manualmente.
- **No reemplazar por:** Yup, Joi, Valibot, ni validación manual.

---

### 8. Formularios: react-hook-form

**Decisión:** `react-hook-form` como librería de gestión de formularios.

**Razones:**
- Performance: re-renders mínimos al no usar estado controlado.
- Integración directa con Zod vía resolvers.
- API consistente que el LLM puede seguir de forma predecible.

**Consecuencias:**
- Formularios siguen la estructura de `component-contracts.md` (props `initialData`, `onSuccess`, `onCancel`).
- **No reemplazar por:** Formik, formularios controlados manuales con `useState`.

---

### 9. Testing: Vitest + React Testing Library + MSW

**Decisión:** Vitest para unit/component tests, MSW para mocking de API, Playwright reservado para E2E futuros.

**Razones:**
- Vitest es compatible con el ecosistema Vite/Next.js, más rápido que Jest para proyectos TypeScript.
- MSW intercepta llamadas HTTP reales — las pruebas testean el código real sin depender del backend Django.
- React Testing Library fuerza pruebas desde la perspectiva del usuario, no de implementación interna.

**Consecuencias:**
- Cobertura mínima: 100% en lógica financiera, 80% en componentes shared.
- Nunca usar IDs de producción en tests — siempre datos de prueba aislados.
- **No reemplazar por:** Jest, Cypress para unit tests, ni `msw` por `axios-mock-adapter`.

---

### 10. Arquitectura Modular: Feature-Sliced Design

**Decisión:** Feature-Sliced Design (FSD) como metodología de organización del código.

**Razones:**
- Delimita explícitamente qué puede depender de qué — evita el acoplamiento circular entre módulos.
- Escalable: cada feature es autónomo y puede desarrollarse en paralelo.
- Compatible con vibe coding: el LLM puede generar código de un feature sin necesitar contexto de otros.

**Consecuencias:**
- Un feature no importa internals de otro feature. Si necesita código compartido, se promueve a `/shared`.
- Los barrel exports (`index.ts`) son el contrato público de cada feature.
- `app/` solo contiene layout y renderizado de página — la lógica vive en `features/`.
- **No reemplazar por:** arquitectura por tipo de archivo (hooks/, components/, utils/ globales).
